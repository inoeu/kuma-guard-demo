# クマ早期警戒・対応支援システム サイト仕様・設計書

作成日: 2026-06-05  
対象: `bear-early-warning-demo`  
状態: 自治体営業・実証提案向けの静的デモサイト

## 1. 目的

本サイトは、自治体向けに「クマ出没リスクを早期に検知し、職員が人間承認を行ったうえで住民通知・現場出動につなげる」運用イメージを説明するための営業デモである。

単なる紹介ページではなく、自治体職員が実際に使う危機管理ダッシュボードに近い画面として見せることを目的とする。

主な訴求点は以下。

- AI検知だけで通知せず、人間承認を必須にする
- 検知、確認、通知、出動、記録の流れを一画面で説明できる
- 住民通知、防災無線、職員SMS、猟友会、観光施設などの連絡先を運用に組み込める
- 議会・予算説明向けに費用対効果、導入ステップ、提案資料の説明までつなげられる

## 2. 想定利用者

### 営業・提案側

- 自治体へ実証実験や導入提案を行う営業担当
- 現地ヒアリングを行う事業責任者
- 補助金、議会説明、予算説明に関わる担当者

### 自治体側

- 危機管理課
- 農林課
- 観光課
- 消防本部
- 学校・観光施設の連絡担当
- 猟友会、巡回隊など現場対応者
- 首長、部長、議会説明を受ける意思決定者

## 3. サイトの位置づけ

現時点では本番システムではなく、静的なフロントエンドデモである。

現在できること。

- サンプルの出没アラート表示
- アラートの選択、絞り込み
- 地図上の候補地点表示
- AI信頼度、検知根拠、推奨対応の表示
- 固定カメラ、ドローン、熱源画像のAI判定デモ表示
- 画像AI判定に対する職員確認、現地確認、誤検知学習への振り分け
- 通知承認、出動要請、誤検知記録のUI上の状態変更
- 通知先の選択
- 巡回班候補の選択
- 対応履歴タイムラインの疑似更新
- 導入提案、費用対効果、導入ステップ、提案資料の表示

現在できないこと。

- 実際のカメラ、センサー、ドローンとの接続
- 実際のLINE、防災無線、SMS、メール送信
- ユーザー認証、権限管理
- データベース保存
- 監査ログの永続保存
- GISや実地図APIとの連携
- 実AIモデルによるリアルタイム推論
- ドローン自動飛行、DIPS申請、飛行計画管理

## 4. 主要画面構成

### 4.1 サイドバー

左側に常設するメニュー。

メニュー項目:

- 今日の判断
- アラート確認
- 監視マップ
- 現場連携
- 導入提案

役割:

- 操作対象を5つに絞り、営業先に説明しやすくする
- 「アラート確認」には承認待ち件数を表示する
- 導入提案を押すと費用対効果セクションへ移動する

### 4.2 トップバー

上部に現在の警戒状態を表示する。

表示項目:

- 警戒レベル
- 市街地接近リスク
- 日付、時刻
- 気象情報
- 初動目標
- 通知件数
- 操作者表示

### 4.3 対応操作バー

今回のUI改善で追加した最重要操作領域。

表示項目:

- 対象アラートのレベル
- アラート名
- 住宅地までの距離
- 初動目標までの残り時間
- AI信頼度
- 通知宛先数
- 巡回班候補

操作:

- 通知承認
- 出動要請
- 提案説明

設計意図:

- 初期表示ですぐに押せる操作を見せる
- スクロールしても対応判断の中心を見失いにくくする
- スマホ幅でも通知承認、出動要請、提案説明が見えるようにする

### 4.4 状況カード

現在の状況を3カードで表示する。

- 今すぐ承認
- 最高リスク
- 次の操作

設計意図:

- 数字を見せすぎず、営業デモ中に見るべき情報を絞る
- 初期画面の認知負荷を下げる

### 4.5 次に判断する内容

対象アラートに対して、職員が何を判断するかを表示する。

表示項目:

- この画面で決めること
- 対象アラート名
- 推奨対応
- 映像確認、通知承認、巡回隊への出動要請のステップ
- 現在の通知・操作状態

### 4.6 承認待ちアラート

アラート一覧を表示する領域。

機能:

- すべて、承認待ち、高リスク、現地確認で絞り込み
- アラートカード選択
- 選択中アラートの強調表示
- AI信頼度、検知元、距離、初動目標を表示

### 4.7 対応判断パネル

選択中アラートの詳細と判断操作を表示する。

表示項目:

- サムネイル
- 承認状態
- 検知時刻
- 検知場所
- 検知機器
- AI信頼度
- 検知タイプ
- 天候・環境
- 推奨対応
- 担当部署、通知チャネル
- 備考
- 対応ステップ
- 通知文プレビュー

操作:

- 住民通知を承認
- パトロール出動要請
- 誤検知として記録

### 4.8 監視マップ

山間部、集落、道路、センサー、出没候補を重ねた疑似地図。

機能:

- 地図 / 航空写真の切り替え
- 倍率操作
- 地図マーカー選択
- 選択中アラートの強調

現時点では実地図APIではなく、SVGベースの疑似マップである。

### 4.9 AI画像判定

固定カメラ、ドローン、熱源画像からAIが検出した候補を表示する。

表示項目:

- 入力種別
- 検知元
- 検知フレーム
- 検知位置
- AIモデル名
- 検出枠
- 信頼度
- 分類候補
- 職員確認状態

操作:

- クマとして確認
- 現地確認へ回す
- 学習キューへ回す

設計意図:

- AIは候補検知に限定し、通知や出動の最終判断は職員が行うことを明確にする
- ドローンは常時監視ではなく、通報後や許可範囲内の現地確認支援として説明する
- 誤検知を学習データとして扱い、改善サイクルを営業時に説明できるようにする

### 4.10 現場連携・運用情報

下部に以下を表示する。

- AI画像判定
- 証拠・記録
- 対応履歴・タイムライン
- 通知先
- 巡回班候補
- 機器稼働状況

### 4.11 自治体営業用ビュー

営業・予算説明に使う下部セクション。

タブ:

- 実証計画
- 費用対効果
- 導入ステップ
- 提案資料

目的:

- 運用デモから、導入提案・予算説明に自然につなげる
- 現場担当だけでなく、意思決定者にも価値を説明する

## 5. 主要操作フロー

### 5.1 通常対応フロー

1. 職員が「今日の判断」を開く
2. 承認待ちアラートを確認する
3. AI画像判定で検出枠、信頼度、分類候補を見る
4. 職員がクマ候補、現地確認、誤検知学習のいずれかを選ぶ
5. 通知先と巡回班候補を確認する
6. 通知承認を押す
7. 必要に応じて出動要請を押す
8. タイムラインに操作履歴が追加される

### 5.2 誤検知記録フロー

1. アラートを選択する
2. 検知根拠と備考を確認する
3. 誤検知として記録を押す
4. ステータスが誤検知記録済みに変わる
5. タイムラインに学習キュー追加として記録される

### 5.3 営業説明フロー

1. 初期画面で検知から承認までの運用を説明する
2. 通知承認、出動要請を実際に押して状態変化を見せる
3. 提案説明を押す
4. 導入提案セクションの費用対効果を表示する
5. 実証計画、導入ステップ、提案資料へ切り替えて説明する

## 8. 非機能要件

### 8.1 ユーザビリティ

- 初期表示で、現在のリスクと次の操作が分かること
- 主要ボタンがスクロール下に埋もれないこと
- スマホ幅でも通知承認、出動要請、提案説明が押せること
- 情報量を増やしすぎず、営業説明中に迷わないこと

### 8.2 レスポンシブ

対応幅:

- デスクトップ: 1280px以上を主対象
- タブレット: 1180px以下で1カラム寄りに整理
- スマホ: 390px幅で横はみ出しなしを確認

### 8.3 アクセシビリティ

- ボタンはネイティブの `button` を使用
- 主要セクションには `aria-label` を設定
- フォーカス時のアウトラインを表示
- 色だけでなくテキストでも状態を示す

### 8.4 セキュリティ

現時点では外部送信を行わない静的デモのため、実通知や個人情報送信は発生しない。

本番化時に必要な対応:

- ログイン認証
- ロール別権限
- 操作監査ログ
- 通知送信前の二重確認
- 個人情報、位置情報、通知履歴の保護

### 8.5 パフォーマンス

現時点ではReact単体の静的ページであり、外部API呼び出しはない。

操作履歴はブラウザの `localStorage` に保存する。これは営業デモ用の簡易保存であり、本番の監査ログではない。

本番化時に注意する点:

- 地図API、センサーAPI、通知APIの遅延
- アラート件数増加時の一覧描画
- 画像・動画証拠の読み込み
- ドローン映像、熱源画像、固定カメラ画像の推論遅延
- タイムラインログのページング

## 9. 技術構成

### 9.1 フロントエンド

- React
- Vite
- lucide-react
- CSS Modulesではなく通常CSS

### 9.2 ビルド

コマンド:

```bash
npm run build
```

出力:

```text
dist/
```

`vite.config.js` で `base: './'` を指定しているため、`dist/index.html` を直接開く場合にもアセット参照が相対パスになる。

### 9.3 ローカル確認

コマンド:

```bash
npm run preview -- --host 127.0.0.1 --port 4174
```

URL:

```text
http://127.0.0.1:4174/
```

注意:

`127.0.0.1` は起動中のMac自身を指す。確認用サーバーが止まっている場合、Chromeでは `ERR_CONNECTION_REFUSED` になる。

## 10. UI設計方針

### 10.1 全体トーン

自治体危機管理ツールとして、派手な広告サイトではなく、実務で使う運用ダッシュボードの見え方を優先する。

方針:

- 暗色ベース
- 赤、黄、緑でリスクと操作を区別
- カードの角丸は控えめ
- 情報密度は高めだが、初期画面は判断に必要な情報へ絞る
- 主要操作は強いコントラストで表示

### 10.2 初期画面の優先順位

1. 現在の警戒レベル
2. 対象アラート
3. 通知承認、出動要請、提案説明
4. 承認待ち件数
5. 検知根拠、AI信頼度
6. 詳細地図、履歴、通知先、提案資料

### 10.3 営業デモで見せる流れ

営業時は以下の順で説明する。

1. 「AIが候補を出す」
2. 「職員が人間承認する」
3. 「通知・出動に進む」
4. 「操作履歴が残る」
5. 「費用対効果と導入工程まで説明できる」

## 13. 受け入れ基準

営業デモとしての受け入れ基準:

- ページが白画面にならない
- 初期表示で対象リスクと主要操作が分かる
- 通知承認、出動要請、提案説明が反応する
- スマホ幅で横はみ出しがない
- 導入提案の費用対効果を説明できる
- 本番システムではなくデモであることが説明できる

実証実験版としての受け入れ基準:

- 実アラートまたは実証用APIからデータ取得できる
- 操作履歴が保存される
- 通知送信はテスト環境で確認できる
- 権限別に操作を制限できる
- 月次レポートの元データを出力できる
